# V2.1 — DreamAgent vs Vector-Retrieval Baseline

**Date:** 2026-05-27
**Snapshot:** night-7 adapter from the V1 Pass 3 chained-training drill
**Base model (both systems):** `mlx-community/Meta-Llama-3.1-8B-Instruct-4bit`
**Probe sets:** 48 personal-recall + 10 cross-memory reasoning
**Top-k for baseline:** 5

## Headline result (and a partial retraction of an earlier claim)

| Probe set | DreamAgent | Retrieval baseline | Δ |
|---|---|---|---|
| `personal_recall` | 75.0% (36/48) | 68.8% (33/48) | **+6.2pp** |
| `cross_memory_reasoning` | 90.0% (9/10) | 90.0% (9/10) | **+0.0pp** |

Baseline mean latency: personal 2,449 ms, cross-memory 2,959 ms (including embed + top-k search + generate).
DreamAgent mean latency from `query_latency`: ~1,300 ms.

## The retraction

The V1 benchmark suite reported a +60pp cross-memory-reasoning advantage of DreamAgent over the base model. That number is real but compared against the wrong baseline — base model with no memories at all. The fair comparison is base model **with retrieval**, which performs identically to DreamAgent on the cross-memory probe set we authored.

The PAPER's §10.5 explicitly listed this as a falsifiable claim:

> *"A retrieval baseline (mem0 with a strong agent harness) scoring > 50% on our cross-memory probe set would weaken the parametric-advantage claim."*

The simple vector-retrieval baseline scored 90%. The cross-memory parametric-advantage claim is therefore weakened — perhaps falsified, depending on how one reads the equality.

## What we still observe

1. **Personal recall: DreamAgent wins by 6.2pp** on the 48-probe set. Likely because the dreamed adapter can recall a specific memory's content directly, while the retrieval baseline depends on top-k surfacing the relevant memory (k=5 of 50 ≈ 90% recall ceiling on the retrieval step alone, before generation noise).
2. **Cross-memory reasoning: parity.** When the retrieval step surfaces all required memories (which our 10-probe set largely permits — most probes require 2-3 memories and top-5 retrieval gives the model headroom), the base model can synthesize across them in context. Parametric memory provides no advantage in this regime.
3. **Latency: roughly comparable.** Retrieval baseline is ~2.5-3s including the embed + search step; DreamAgent is ~1.3s mean. The win is modest and config-dependent.

## What this means for the V2 thesis

The original V2 framing — "the dreamed model is uniquely capable of cross-memory reasoning that retrieval cannot deliver" — is too strong. A better framing:

**DreamAgent and retrieval are complementary, not competitive.** Specifically:

- **DreamAgent uniquely offers**: privacy (weights, no index), survival of vector-retrieval misses, the ability to pivot to a different host agent without reindexing.
- **Retrieval uniquely offers**: trivial deletion (drop the chunk), instant updates (no nightly fine-tune), recall ceiling determined by k, not by training capacity.
- **Composition wins**: both running in parallel — retrieval for hot, recent, mutable memories; DreamAgent for consolidated, durable, "who is this user" knowledge.

The V2.2 work item "mem0 + DreamAgent composition" was already in the roadmap. After this result, it becomes the most important V2.2 deliverable, not an optional polish.

## What the experiment did NOT test

- mem0 specifically (we used a simpler `sentence-transformers` baseline)
- Letta with filesystem tools — Letta's own benchmark notes [\[1\]](https://www.letta.com/blog/benchmarking-ai-agent-memory) show filesystem-based agents beat specialized memory systems on LoCoMo, suggesting our simple retrieval baseline is probably weaker than what a serious agent harness would deliver
- Adversarial probe sets where retrieval explicitly fails (e.g., questions that don't trigger semantic similarity but require parametric knowledge)
- Very large memory corpora (50 memories is small; retrieval's k=5 covers 10% of the corpus — at 10,000 memories, k=5 covers 0.05% and the recall ceiling drops)
- The privacy axis (which is structural, not empirical)
- The "host agent independence" axis

A more thorough V2.1 would extend the comparison along these dimensions. We have the budget for one expansion before V2.2 work begins — most likely: **adversarial probe set where retrieval is forced to fail**, e.g. by asking questions whose semantic embedding doesn't match the relevant memory's content.
