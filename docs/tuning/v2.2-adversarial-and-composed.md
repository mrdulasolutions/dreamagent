# V2.2 — Adversarial probes + three-way head-to-head

**Date:** 2026-05-27
**Snapshot:** night-7 adapter from the V1 Pass 3 chained-training drill
**Base model (all three systems):** `mlx-community/Meta-Llama-3.1-8B-Instruct-4bit`
**Top-k for retrieval baseline:** 5

## What we tested

V2.1 measured **DreamAgent vs retrieval** on personal-recall and
cross-memory probes and found parity on cross-memory reasoning. The
honest question that opens: **does parametric memory have ANY domain
where it empirically wins, or is it always at-best matched by retrieval?**

V2.2 tests two new things:

1. **Adversarial probes** (`benchmarks/probes/adversarial_retrieval.jsonl`)
   — 15 questions designed with low semantic similarity to their target
   memories. The hypothesis: retrieval should miss; DreamAgent should
   still answer.

2. **Composed system** — query DreamAgent AND retrieval, then ask the same
   base model to reconcile. Tests the V2.2 production architecture.

The benchmark (`benchmarks/three_way.py`) runs all three systems
(DreamAgent / retrieval / composed) across all three probe sets
(personal / cross-memory / adversarial) — 73 probes × 3 systems = 219
generations, plus 73 reconciliation calls for the composed mode.

## Adversarial probe design

Each adversarial probe is constructed so:

- The target memory exists in the training set (`mem_fix_*`)
- The question's surface form has **deliberately low embedding similarity** to the memory
- An answer is still derivable for an agent that has the memory in weights

Five techniques:

| Technique | Example |
|---|---|
| Counterfactual framing | Memory: "user prefers Apache/MIT licenses." Q: "Should I recommend GPL-3.0?" |
| Different domain vocabulary | Memory: "user works at Mrdula Solutions." Q: "Who pays the user's salary?" |
| Symbol-vs-word | Memory: "no emojis in commit messages." Q includes an actual 🎉 character. |
| Indirect reference | Memory: "primary project is DreamAgent." Q: "The user mentioned dreaming of better memory consolidation. What's their project called?" |
| Multi-hop with arithmetic | Memory: "anniversary is August 18th." Q: "If I remind them 3 days before, when?" |

Full probe set: [`benchmarks/probes/adversarial_retrieval.jsonl`](../../benchmarks/probes/adversarial_retrieval.jsonl). Each probe declares its target memory IDs explicitly; reviewers can verify the constructions.

## Interpretation rubric (pre-registered)

To prevent post-hoc rationalization, we pre-register what we'd say given
each possible outcome before seeing the numbers:

### If DreamAgent > Retrieval on adversarial by ≥ 10pp

→ The parametric advantage is real but conditional: when retrieval's
semantic-similarity assumption fails, parametric memory has a measurable
edge. This is the V2 thesis vindicated in a narrow but defensible domain.

### If DreamAgent ≈ Retrieval on adversarial (within ±5pp)

→ Either retrieval is more robust than expected (the base model's
generation step can recover from imperfect retrieval) or our adversarial
probes are weaker than designed. We'd commit to a stronger adversarial
set in V2.3 before claiming parametric superiority.

### If DreamAgent < Retrieval on adversarial

→ Our adversarial probes were actually easy. Retrieval is more robust
than we believed. The honest V2 positioning collapses to "DreamAgent
is privacy-strict and host-independent; retrieval matches or beats it
on capability under all measured conditions."

### If Composed > both individual systems by ≥ 5pp on any probe set

→ The V2.2 composition story is empirically justified. mem0 + DreamAgent
running together is the recommended production architecture. The
composition cookbook ([`examples/08-mem0-plus-dreamagent/`](../../examples/08-mem0-plus-dreamagent/))
becomes a meaningful artifact.

### If Composed ≤ max(DA, retrieval)

→ Composition doesn't help; the reconciliation step costs more than it
adds. The production recommendation becomes "pick whichever is best
for your use case, don't bother running both."
